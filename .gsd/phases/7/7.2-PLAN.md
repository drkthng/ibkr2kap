# Plan 7.2: FIFO Engine Integration & Runner

## Objective
Integrate the FIFO Engine to process the entire database chronologically, updating accounts and handling all parsed trades.

## Context
- .gsd/SPEC.md
- src/ibkr_tax/services/fifo.py (from Plan 7.1)
- .gsd/phases/7/RESEARCH.md

## Tasks

<task type="auto">
  <name>Build FIFO Runner Service</name>
  <files>src/ibkr_tax/services/fifo_runner.py</files>
  <action>
    Create a `FIFORunner` class/functions that:
    - Fetches all `Trade`s for an `Account`, ordered strictly by `settle_date` ASC, then `id` ASC.
    - Groups them by `symbol`.
    - Feeds them iteratively into the `FIFOEngine` from Plan 7.1.
    - Commits the resulting `FIFOLot` and `Gain` objects to the database.
    - Adds idempotency: if the trades were already processed (or if we need to rerun), it should be able to clear existing Lots/Gains for the account and re-run cleanly.
  </action>
  <verify>pytest -k "test_fifo_runner"</verify>
  <done>FIFO runner can process an entire account's trades idempotently and save results to the DB.</done>
</task>

<task type="auto">
  <name>Create FIFO Runner Integration Tests</name>
  <files>tests/test_fifo_runner.py</files>
  <action>
    Write integration tests for `fifo_runner.py`:
    - Setup an account with multiple trades across different symbols in the database.
    - Run the `FIFORunner`.
    - Verify that `FIFOLot` and `Gain` records are accurately persisted.
    - Verify idempotency (running twice produces the exact same DB state, or clears and recreates it cleanly).
  </action>
  <verify>pytest tests/test_fifo_runner.py -v</verify>
  <done>Integration tests prove that DB state is correctly updated and runner is idempotent.</done>
</task>

## Success Criteria
- [ ] Entire database of trades can be processed into matching lots and gains.
- [ ] Rerunning the engine is completely safe and idempotent.
