# Plan 6.2: End-to-End Import Pipeline

## Objective
Connect the existing XML/CSV parsers with the new repository layer to create a seamless, end-to-end data import pipeline.

## Context
- .gsd/SPEC.md
- src/ibkr_tax/services/flex_parser.py
- src/ibkr_tax/services/csv_parser.py
- src/ibkr_tax/db/repository.py

## Tasks

<task type="auto">
  <name>Create Import Pipeline Service</name>
  <files>src/ibkr_tax/services/pipeline.py</files>
  <action>
    Create `pipeline.py` exposing a main `run_import(file_path: str, session: Session, file_type: str = "xml")` function.
    The function should invoke `flex_parser` or `csv_parser` based on `file_type`.
    It should then pass the parsed `AccountSchema`, `TradeSchema`, and `CashTransactionSchema` lists to the repository functions.
    Construct and return a summary dictionary with counts of records parsed vs. actually inserted for each type to give visibility.
  </action>
  <verify>pytest -k "test_pipeline" -v</verify>
  <done>Data flows correctly from raw file to database insertion returning metrics.</done>
</task>

<task type="auto">
  <name>Verify End-to-End Import</name>
  <files>tests/test_pipeline.py</files>
  <action>
    Create integration tests using the existing sample XML/CSV files (or small test fixtures).
    Run `run_import` on a sample file with a test database session.
    Assert that the correct numbers of Accounts, Trades, and CashTransactions are present in the database post-import.
  </action>
  <verify>pytest tests/test_pipeline.py -v</verify>
  <done>Pipeline successfully ingests a full sample file into the test DB.</done>
</task>

## Success Criteria
- [ ] Single entry-point function `run_import` orchestrates parsing and db insertion.
- [ ] Existing XML/CSV samples can be ingested end-to-end.
- [ ] Pipeline reports accurate insertion metrics.
